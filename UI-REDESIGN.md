# UI redesign — github-native edition

## Why

The v1 editorial UI (warm paper, Fraunces serif, narrow column, rust accent) is polished but the author rejected it as a fit for a dev tool. After a five-direction brainstorm (mockups at `mockups/`: terminal · github · brutalist · arcade · notebook) the **github-native** direction was picked.

The redesign goal is to make `quizz take` feel like an extension of the PR review surface — the place dev work already happens — instead of an editorial aside. Same engine, same flow, new clothes.

## Scope

**In scope.** The browser surface served by `src/quizz/server/`. Three states of a single-page app:

1. **Question state** — the form the author fills out
2. **Results state** — post-submit, scored locally, not yet published
3. **Published state** — after the author clicks publish, comment is live on the PR

**Out of scope.** Engine, CLI, comment markdown (`src/quizz/comment/render.py`), grading logic, question generation, dark mode, i18n, print styles. Backend is *almost* out of scope — one exception: `POST /publish` gains a `comment_url` field in its response (see Risks). This is otherwise a UI-only swap.

## Decisions

Three questions raised during the mockup review were answered:

| # | Question | Decision |
|---|---|---|
| 1 | Use the GitHub "files" metaphor for questions? | **No.** Keep the card/header chrome, but label cards `Question 1` / `Question 2`. Drop the fake `.mcq` / `.open` extensions and the "Answered" viewed-checkbox. Questions aren't files; the metaphor ages badly. |
| 2 | Render a rich posted-comment preview in `published.html`? | **No.** The published view shows a success banner and a *link* to the posted comment. Rich per-question breakdown stays in the local results view. The PR comment itself remains a simple markdown table (handled by `src/quizz/comment/render.py`, no change). |
| 3 | Three-band score ring (red / orange / green)? | **No.** Single neutral ink color (`#1f2328`) always. The score is data, not a verdict. "Failing is the point" doesn't survive a traffic light. |

Two more decisions made implicitly by the mock-review:

| # | Question | Decision |
|---|---|---|
| 4 | Should the fake top bar say "GitHub"? | **No** — it says **quizz** with the same dark-bar visual chrome. Pretending to be GitHub is dishonest and breaks the moment the user looks at the URL. |
| 5 | Keep the sidebar? | **Yes.** It carries the progress / score / timeline signal and matches the GitHub PR layout that anchors the whole aesthetic. Stacks above content on mobile (already wired in the mocks). |

## Architecture

No new code paths, no new endpoints. The FastAPI app at `src/quizz/server/app.py` keeps its three routes (`GET /`, `POST /submit`, `POST /publish`) and its three template substitutions (`__PR__`, `__PR_URL__`, `__QUIZ_JSON__`). The redesign is a full replacement of three files:

```
src/quizz/server/assets/
├── index.html       ← rewrite: github-native shell (topbar, repo header, tabs, sidebar, sticky bar)
├── styles.css       ← rewrite: light theme, Primer-inspired tokens, JetBrains Mono for code
└── quiz.js          ← rewrite render functions: questions / results / published
                       (keep submit + publish wiring; only the DOM the renders build is new)
```

`mermaid.min.js` stays as-is. Mermaid node/edge styling is restyled inside `styles.css` from "blueprint cyan on textured paper" to "ink-on-white with blue accent" — matches the new palette.

## The three states

### State 1 — Question (`render_questions`)

A single-column main area inside a max-1280px container, with a 296px right sidebar.

**Per-question card.** Same chrome as the mockup — a bordered card with a tinted `__head` strip and an expanded `__body`. The head shows: number (`Question 1`), type pill (`MULTIPLE CHOICE`), and a small dot indicating answered/unanswered. No `.mcq` extension. No viewed checkbox.

**Inputs.**

- MCQ / TF: radio-style rows, blue selection (`#0969da` border + `#ddf4ff` fill), letter chip (A/B/C/D) on the left.
- Mermaid: 2×2 grid of cards, each containing a `<div class="mermaid">` block rendered by `mermaid.js`. Selected card gets a blue border + checkmark in the corner.
- Open: textarea inside a 1px-bordered card, focus-ring on blue. Character counter below.

**Sidebar.** Progress dots (filled per answered question), question list with type pills, "Generated by claude-sonnet-4-6" attribution, diff scope (`6 files · +241 −106`).

**Sticky review bar.** Status message + `Submit quiz` primary button (green, GitHub-style). Wraps on mobile (already in the mocks).

### State 2 — Results (`render_results`)

Same shell. Main area swaps to:

1. **Summary card.** Score ring on the left (CSS conic-gradient, single ink color), score caption on the right (`Scored locally · 3 of 5 right`), a 5-pip scorecard strip (one pip per question, ✓/✗/~ glyphs in green/red/orange tints — these glyph colors stay, only the *ring* goes neutral).
2. **Per-question result cards.** Same card chrome as questions, but the left border is colored (green/red/orange) and the head shows the score (e.g. `100 / 100 · correct`). Body shows the user's answer highlighted in pass/fail tint, the correct answer below if different, and for open questions, an LLM-feedback block (purple left border, avatar with model name).
3. **Diagrams in result mode.** Show only user's pick + the correct diagram side-by-side. A "show all 4" link expands the rest. Keeps the result view scannable.

**Sidebar.** Swaps to: Score block (big number + thin bar) → Per-question list with ✓/✗ icons → Graded by (model attribution) → Visibility (`Private to you — not posted yet`).

**Sticky bar.** Two buttons: ghost `Discard` + primary green `Publish to PR`. Mobile-stacked.

### State 3 — Published (`render_published`)

Same as Results, with three additions:

1. **Success banner** at the top of the main area: green-bordered card, checkmark icon, "Posted to PR · ~4 seconds ago" + a `View comment` link out to the actual GitHub comment URL.
2. **Sidebar Visibility block replaces** with **Timeline block** (Quiz generated → Answered → Graded → Published, last one highlighted with a green ring).
3. **Sticky bar** flips to green-tinted: `[✓ Published] Scorecard live on #142.` with `Delete comment` (ghost) and `Open on GitHub` (external link).

**No comment preview.** The fancy mocked-up posted-comment is *not* part of the real published view — the user clicks the link and sees their actual comment on GitHub.

## Component inventory

To keep the rewrite reviewable, the CSS should be organized by these named sections (single file, but commented dividers):

```
/* tokens          */  CSS custom properties (Primer-ish: --fg, --fg-mute, --blue, --green, ...)
/* base            */  reset, typography, links, body bg
/* topbar          */  dark "quizz" bar (decorative chrome)
/* repohead        */  breadcrumb, PR title, meta row, tab strip
/* container       */  main + sidebar grid
/* card            */  shared question/result card chrome
/* inputs/mcq      */  radio rows, TF pills
/* inputs/mermaid  */  diagram pick cards + mermaid node/edge overrides
/* inputs/open     */  textarea
/* sidebar         */  progress, score, question list, timeline
/* summary         */  results score ring + scorecard strip
/* feedback        */  LLM feedback bubble (purple)
/* banner          */  published success banner
/* reviewbar       */  sticky bottom action bar (3 states: submit / publish / published)
/* responsive      */  mobile breakpoints
```

Roughly 600–800 lines target. Current `styles.css` is 860 lines; this should land comparable.

## JS state flow

```
window.QUIZ + window.PR_URL  →  renderQuestions()
                                       │  submit click
                                       ▼
                            POST /submit  →  Results JSON
                                       │
                                       ▼
                                renderResults(results)
                                       │  publish click
                                       ▼
                            POST /publish  →  { ok: true, comment_url? }
                                       │
                                       ▼
                                renderPublished(results, comment_url)
```

Three render functions, one wiring function. The current `quiz.js` already has the submit/publish wiring — we keep it. Only the DOM each render builds is new.

**Important quirk to preserve.** `app.py` already substitutes the quiz JSON into `window.QUIZ` at template render time — no `fetch` needed for the initial quiz. Same trick stays.

**Comment URL.** `POST /publish` currently returns `{ ok, total_score }`. We need it to also return `comment_url` for the success banner's "View comment" link. That's a one-line backend change in `app.py` — flagged as the single back-end touch in this redesign.

## Mermaid styling

The current stylesheet renders mermaid as blueprint diagrams (cyan strokes on dotted-grid blue paper) — that aesthetic is leaving with the editorial UI. New treatment:

- Card background: `--bg-subtle` (#f6f8fa)
- Node stroke: `--blue` (#0969da), 1.5px
- Selected card: blue border + `--blue-bg` (#ddf4ff) fill
- Edge stroke: `--fg` (#1f2328)
- Node label font: JetBrains Mono, 12px, `--fg`
- 429 / error edges: `--red` (#cf222e)

Keeps the structural identity but matches the rest of the UI.

## Responsive behavior

Already battle-tested in the mocks at 1440 / 720 / 390 widths. Carry the patterns over:

- Tabs row: `overflow-x: auto`, hidden scrollbar, `flex-shrink: 0` on tabs (matches real GitHub mobile)
- Sticky bar: `flex-wrap: wrap`, message stacks above buttons at ≤700px
- Sidebar: stacks above main content at ≤900px (`grid-template-columns: 1fr`)
- Banner CTA: drops to its own row at ≤600px

## Out of scope (explicit)

- Dark mode
- Comment markdown changes (`src/quizz/comment/render.py` unchanged)
- New question types
- Engine/grading changes
- Real GitHub Primer CSS dependency (we use our own tokens that *match* Primer values, no npm dep)
- Accessibility audit beyond focus styles + semantic HTML (parity with current, not WCAG AAA)
- Server-side rendering of the posted-comment preview

## Risks & open implementation decisions

1. **`POST /publish` response shape.** Adding `comment_url` is the one back-end change. Need to confirm `src/quizz/ghio/` exposes the URL of the comment after creation — if it doesn't, the banner falls back to a generic "View on GitHub" link to the PR.
2. **Mermaid render timing.** Mermaid sometimes renders nodes wider than their container, causing horizontal overflow. The current code uses `mermaid.run()` after each render — we'll need to call it again after `renderResults` / `renderPublished` if any mermaid blocks are still in the DOM (results state shows the user's diagram pick + correct one).
3. **The fake topbar / repo header may feel performative.** It's chrome that mimics GitHub *without* being it. Acceptable for v1 but worth re-evaluating after first use — if it reads as cute-but-empty, we strip it to just the breadcrumb + PR title.

## Testing

- The existing Playwright test driver (`/tmp/quizz-battle-test/run.py`) is the template — extend it to drive the real FastAPI app (not just static mocks) and exercise the question → results → published transitions. Replaces / extends any existing screenshot tests in `tests/`.
- Manual: `quizz take` against a real PR, walk all three states, publish, verify the comment lands.
- Existing engine/server tests should still pass — only UI changes.
